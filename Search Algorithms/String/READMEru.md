# Сравнение алгоритмов поиска подстроки: Boyer-Moore, Rabin-Karp, KMP, Z-алгоритм, Манакер, наивный поиск

## Основные отличия

| Алгоритм                | Временная сложность                | Пространственная сложность | Применимость и замечания |
|:-----------------------:|:----------------------------------:|:-------------------------:|:------------------------|
| Наивный поиск           | O((n−m+1)m)                        | O(1)                      | Очень прост, не требует доп. памяти, но медленный на больших данных. Хорош для обучения и малых задач. |
| Rabin-Karp              | O(n + m) (ср.) / O(nm) (худш.)     | O(1)                      | Быстр в среднем случае за счет хеширования, легко адаптируется для поиска нескольких шаблонов. Может вырождаться в наивный в худшем случае. |
| KMP (Кнута-Морриса-Пратта) | O(n + m)                      | O(m)                      | Всегда быстр, требует память для префикс-функции (lps). Отлично подходит для больших текстов, если память не критична. |
| Boyer-Moore             | O(n + m) (худш.), O(n/m) (ср.)     | O(m + k)                  | Очень быстр на практике для длинных шаблонов и больших алфавитов. Использует доп. память для таблиц. Лучший выбор для поиска одного шаблона в больших текстах. |
| Z-алгоритм              | O(n)                               | O(n)                      | Отлично подходит для поиска всех вхождений шаблона. Полезен для препроцессинга строк и сложных задач. |
| Алгоритм Манакера       | O(n)                               | O(n)                      | Специализирован для поиска всех палиндромных подстрок за линейное время. Не для обычного поиска подстроки. |

*Здесь n — длина текста, m — длина шаблона, k — размер алфавита.*

---

## Подробнее об алгоритмах

- **Наивный поиск:**
    - Сравнивает шаблон со всеми позициями в тексте.
    - Не требует предварительной обработки или доп. памяти.
    - Медленный на больших данных.
    - Полезен для обучения, тестов и очень маленьких задач.

- **Rabin-Karp:**
    - Использует хеширование для быстрого поиска в среднем случае, если мало коллизий.
    - При коллизиях хешей требуется посимвольное сравнение.
    - Легко расширяется для поиска множества шаблонов.
    - Требует минимум памяти (только для хешей).
    - В худшем случае (много коллизий) — вырождается в наивный (O(nm)).

- **KMP (Кнут-Моррис-Пратт):**
    - Без хешей, строит только префикс-функцию для шаблона.
    - Никогда не делает лишних сравнений, всегда O(n + m).
    - Требует O(m) памяти для префикс-функции.
    - Не очень подходит для поиска многих шаблонов без модификаций.
    - Отличен для длинных повторяющихся текстов.

- **Boyer-Moore:**
    - Строит таблицы "плохой символ" и "хороший суффикс".
    - Сравнивает шаблон справа налево, позволяет делать большие сдвиги по тексту.
    - Очень быстр для длинных шаблонов и больших алфавитов.
    - Использует O(m + k) памяти для таблиц.
    - Лучший выбор для поиска одного шаблона в большом тексте.
    - Менее эффективен для коротких шаблонов или маленьких алфавитов.

- **Z-алгоритм:**
    - Строит массив Z-значений (длины подстрок, совпадающих с префиксом).
    - Полезен для поиска всех вхождений шаблона.
    - Отлично подходит для сложных строковых задач и препроцессинга.
    - Работает за O(n) времени и памяти.

- **Алгоритм Манакера:**
    - Предназначен для поиска всех палиндромных подстрок за линейное время.
    - Не подходит для обычного поиска подстроки.
    - Используется для задач типа "самая длинная палиндромная подстрока".

---

## Где алгоритмы могут "ломаться" (худший случай или потеря эффективности)

- **Наивный поиск**
    - "Ломается" (становится очень медленным) на длинных текстах и/или шаблонах, особенно если текст содержит много повторяющихся символов, совпадающих с началом шаблона.
    - Пример: поиск `"aaaaab"` в `"aaaaaaaaaaaaaaaaaaaab"`.

- **Rabin-Karp**
    - Худший случай при большом числе коллизий хешей (много совпадающих хешей, но строки разные).
    - В этом случае требуется посимвольное сравнение для каждого совпадения — сложность O(nm).
    - При плохой хеш-функции или огромном алфавите вероятность коллизий выше.

- **KMP**
    - Почти никогда не "ломается" по времени, но требует много памяти для длинных шаблонов (O(m)).
    - Неэффективен для поиска множества разных шаблонов — требуется строить отдельную функцию для каждого.
    - Мало пользы для очень коротких шаблонов.

- **Boyer-Moore**
    - Теряет преимущество на очень коротких шаблонах или маленьких алфавитах.
    - Худший случай: текст и шаблон состоят из повторяющихся символов, совпадающих с концом шаблона — минимальные сдвиги.
    - Неудобен для поиска множества шаблонов одновременно.

- **Z-алгоритм**
    - Не очень выгоден для поиска одного шаблона в огромных текстах из-за O(n) памяти.
    - В основном эффективен и устойчив для своих задач.

- **Манакер**
    - Применим только для задач с палиндромами, не для общего поиска подстроки.

---

## Итоги

- **Мало памяти?** — Rabin-Karp или наивный, почти не требуют памяти.
- **Гарантия скорости?** — KMP хорош, особенно для повторяющихся текстов.
- **Много шаблонов?** — Rabin-Karp (с множеством), Z-алгоритм (для продвинутых задач).
- **Один длинный шаблон, большой текст и нужна максимальная скорость?** — Boyer-Moore.
- **Нужны все палиндромные подстроки?** — Манакер (для этого уникален).
- **Наивный** — только для демонстраций и маленьких данных.
