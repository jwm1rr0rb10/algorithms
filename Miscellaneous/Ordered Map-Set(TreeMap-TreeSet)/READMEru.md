# 🔍 Упорядоченная Карта/Множество (TreeMap/TreeSet)

## 📌 Описание Проблемы

**Упорядоченная карта** (часто называемая `TreeMap` в Java, или `std::map` в C++, или `SortedDict` в некоторых библиотеках Python, таких как `sortedcontainers`) и **Упорядоченное множество** (`TreeSet` в Java, `std::set` в C++, или `SortedList` в `sortedcontainers`) — это структуры данных, которые хранят пары "ключ-значение" (для карт) или уникальные элементы (для множеств), поддерживая их в **отсортированном порядке на основе ключей (или самих элементов)**.

В отличие от `HashMap` / `HashSet`, которые обеспечивают среднее время поиска $O(1)$, но не имеют внутренней упорядоченности, Упорядоченные карты/множества гарантируют, что элементы всегда будут перебираться в отсортированной последовательности. Это свойство упорядоченности достигается ценой немного более высокой временной сложности для операций вставки, удаления и поиска по сравнению с хеш-основанными структурами.

---

## 💡 Идея и Подход

Основная идея Упорядоченных карт и множеств заключается в том, что они обычно реализуются с использованием **самобалансирующихся двоичных деревьев поиска (BST)**. Распространенные типы самобалансирующихся BST включают:

* **Красно-черные деревья (Red-Black Trees):** Популярный выбор (используется `TreeMap`/`TreeSet` в Java, `std::map`/`std::set` в C++). Они гарантируют, что дерево остается сбалансированным (высота $O(\log n)$) путем применения вращений и изменений цвета во время вставки и удаления, обеспечивая логарифмическую временную сложность для операций.
* **AVL-деревья (AVL Trees):** Другой тип самобалансирующихся BST, известный своей строгой сбалансированностью.
* **Трипы (Treaps):** Рандомизированная структура данных, которая сочетает свойства двоичных деревьев поиска и куч.

Независимо от конкретного самобалансирующегося BST, фундаментальные принципы таковы:

1.  **Свойство двоичного дерева поиска:** Для любого узла все ключи в его левом поддереве меньше его ключа, а все ключи в его правом поддереве больше его ключа.
2.  **Свойство самобалансировки:** После любой вставки или удаления дерево автоматически выполняет операции (вращения, перекрашивание) для поддержания сбалансированной структуры, предотвращая дегенерацию дерева в связный список (что привело бы к временной сложности $O(n)$ в худшем случае).
3.  **Инфиксный обход (In-order Traversal):** Инфиксный обход BST всегда посещает узлы в отсортированном порядке их ключей. Это свойство позволяет Упорядоченным картам/множествам обеспечивать отсортированную итерацию.

### Ключевые Операции:

* **`put(key, value)` / `add(element)`:** Вставляет новую пару "ключ-значение" (или элемент), поддерживая отсортированный порядок и баланс.
* **`get(key)` / `contains(element)`:** Извлекает значение, связанное с ключом (или проверяет наличие элемента), путем обхода дерева на основе сравнения ключей.
* **`remove(key)` / `remove(element)`:** Удаляет пару "ключ-значение" (или элемент), поддерживая порядок и баланс.
* **`first_key()` / `last_key()`:** Получает наименьший/наибольший ключ в карте/множестве.
* **`lower_bound(key)` / `upper_bound(key)`:** Находит наименьший ключ, который больше или равен (или строго больше) заданного ключа. (Особенности `std::map` в C++)
* **`floor_key(key)` / `ceiling_key(key)`:** Находит наибольший ключ, который меньше или равен (или наименьший ключ, который больше или равен) заданного ключа. (Особенности `TreeMap` в Java)
* **`iterate()`:** Перебирает элементы в отсортированном порядке.

## 🧪 Пример на Python

Встроенные в Python `dict` и `set` основаны на хешировании и **не поддерживают** порядок вставки (для `dict` до Python 3.7; после 3.7 порядок вставки сохраняется, но он все равно основан на хешировании). Для действительно *отсортированных* карт и множеств (на основе значений ключей, а не порядка вставки) обычно используются внешние библиотеки или реализуется BST самостоятельно.

Библиотека `sortedcontainers` является популярным выбором в Python для этой функциональности.

```python
# Возможно, вам потребуется установить ее: pip install sortedcontainers
from sortedcontainers import SortedDict, SortedList

print("--- Пример Упорядоченной Карты (SortedDict) ---")
my_map = SortedDict()

# 1. Вставка
my_map[5] = "apple"
my_map[2] = "banana"
my_map[8] = "cherry"
my_map[1] = "date"
my_map[5] = "grape" # Обновляет значение для ключа 5

print(f"Карта после вставок (отсортированная по ключу): {my_map}")
# Вывод покажет элементы, отсортированные по ключу: SortedDict({1: 'date', 2: 'banana', 5: 'grape', 8: 'cherry'})

# 2. Поиск
print(f"Значение для ключа 2: {my_map.get(2, 'Не найдено')}") # Вывод: banana
print(f"Значение для ключа 7: {my_map.get(7, 'Не найдено')}") # Вывод: Не найдено

# 3. Проверка существования ключа
print(f"Ключ 8 в карте? {8 in my_map}") # Вывод: True

# 4. Удаление
del my_map[2]
print(f"Карта после удаления ключа 2: {my_map}")
# Вывод: SortedDict({1: 'date', 5: 'grape', 8: 'cherry'})

# 5. Итерация (всегда отсортирована по ключу)
print("Итерация по ключам карты:")
for key in my_map:
    print(f"Ключ: {key}, Значение: {my_map[key]}")

# 6. Специфические операции упорядочивания (пример с SortedDict)
print(f"Наименьший ключ: {my_map.keys()[0]}")   # Или my_map.peekitem(0)[0]
print(f"Наибольший ключ: {my_map.keys()[-1]}")   # Или my_map.peekitem(-1)[0]
print(f"Количество ключей меньше или равно 5: {my_map.bisect_right(5)}") # Возвращает индекс, куда 5 можно вставить для поддержания порядка (первый индекс > 5)
                                                                       # Правильное использование для диапазона: my_map.irange(None, 5)

print("\n--- Пример Упорядоченного Множества (SortedList) ---")
my_set = SortedList()

# 1. Вставка (добавление уникальных элементов, отсортированных)
my_set.add(50)
my_set.add(20)
my_set.add(80)
my_set.add(10)
my_set.add(50) # Дубликаты игнорируются

print(f"Множество после добавлений (отсортированное): {my_set}")
# Вывод: SortedList([10, 20, 50, 80])

# 2. Проверка существования элемента
print(f"20 в множестве? {20 in my_set}") # Вывод: True
print(f"70 в множестве? {70 in my_set}") # Вывод: False

# 3. Удаление
my_set.remove(20)
print(f"Множество после удаления 20: {my_set}")
# Вывод: SortedList([10, 50, 80])

# 4. Итерация (всегда отсортирована)
print("Итерация по элементам множества:")
for item in my_set:
    print(item)

# 5. Специфические операции упорядочивания (пример с SortedList)
print(f"Наименьший элемент: {my_set[0]}")
print(f"Наибольший элемент: {my_set[-1]}")
print(f"Количество 50: {my_set.count(50)}")
print(f"Индекс 50: {my_set.index(50)}")
```

## ⏱️ Сложность

Временная сложность для **Упорядоченных карт/множеств**, реализованных с использованием самобалансирующихся BST, является логарифмической.

* **Временная Сложность (Средний и Худший Случай):**
    * **Вставка (`put`/`add`):** $O(\log n)$
    * **Удаление (`remove`):** $O(\log n)$
    * **Поиск (`get`/`contains`):** $O(\log n)$
    * **Нахождение мин/макс (`first`/`last`):** $O(\log n)$ (или $O(1)$, если реализация поддерживает прямые указатели на мин/макс узлы, что распространено).
    * **Запросы диапазона/Итерация:** $O(k + \log n)$, где $k$ — количество элементов в диапазоне. $O(\log n)$ — это время нахождения начала диапазона, а $k$ — на обход элементов.

* **Пространственная Сложность:**
    * $O(n)$
    Структура данных должна хранить все $n$ пар "ключ-значение" (или элементы). Каждый узел в дереве обычно требует больше памяти, чем запись в хеш-таблице, из-за указателей (левый, правый, родительский, цвет) и, возможно, метаданных, связанных с балансом.

## ⚠️ Соображения

* **Компромиссы в Производительности:** Упорядоченные карты/множества предлагают гарантированный отсортированный порядок и $O(\log n)$ в худшем случае, что лучше, чем $O(n)$ в худшем случае для `HashMap`. Однако их $O(\log n)$ в среднем случае обычно медленнее, чем $O(1)$ в среднем случае для `HashMap` из-за накладных расходов на обходы дерева и операции балансировки. Выбирайте на основе того, что более критично: отсортированный порядок или чистая скорость для отдельных операций.
* **Сравнимость Ключей/Элементов:** Ключи (или элементы) должны быть сравнимыми (реализовывать `__lt__` или аналогичный метод в Python), чтобы дерево могло правильно их упорядочить.
* **Накладные Расходы по Памяти:** Узлы в древовидной структуре часто потребляют больше памяти, чем записи в хеш-таблице, из-за необходимости в указателях на дочерние и, возможно, родительские узлы, а также метаданных, связанных с балансом.
* **Запросы Диапазона:** Одно из самых сильных преимуществ упорядоченных карт/множеств — это их способность выполнять эффективные запросы диапазона (например, "найти все элементы между ключом X и ключом Y"), что невозможно эффективно с хеш-основанными структурами.

## 🧭 Применения

Упорядоченные карты и множества незаменимы, когда порядок элементов или эффективные запросы на основе диапазона имеют решающее значение.

* **Базы Данных (Индексирование):** B-деревья (обобщение BST) обычно используются для индексирования баз данных, что обеспечивает эффективный отсортированный поиск и запросы диапазона на больших наборах данных.
* **Планирование/Управление Событиями:** Хранение событий или задач, упорядоченных по времени или приоритету.
* **Пользовательская Сортировка:** Поддержание коллекции элементов, которые всегда должны быть представлены или доступны в отсортированном виде.
* **Поиск Ближайших Элементов:** Эффективный поиск элемента, непосредственно большего или меньшего заданного значения.
* **Запросы Диапазона:** Любой сценарий, где вам нужно получить все элементы в определенном диапазоне ключей/значений (например, "найти всех пользователей, чей балл находится между 100 и 200").
* **Реализация Других Структур Данных:** Могут использоваться в качестве строительных блоков для более сложных структур данных.
* **Автодополнение/Поиск в Словарях:** Может быть эффективным для частичных совпадений или поиска по префиксу.
* **Порядковая Статистика (K-й наименьший/наибольший):** Хотя не напрямую, сбалансированное BST может быть дополнено для эффективного нахождения K-го наименьшего/наибольшего элемента.

## 🧩 Связь с LeetCode

Упорядоченные карты и множества, или базовая концепция сбалансированных BST, часто встречаются в задачах LeetCode, особенно когда:

* Вам нужно поддерживать данные в отсортированном порядке.
* Вам нужно найти элементы, которые "чуть больше" или "чуть меньше" заданного значения (`ceiling`, `floor`).
* Вам нужно выполнять эффективные запросы по диапазону.
* Ограничения задачи включают сложность $O(\log n)$, что указывает на древовидные решения.
* Вам предлагается реализовать структуру данных, которая должна поддерживать порядок (например, пользовательский поток данных, где элементы обрабатываются по порядку).
* Такие задачи, как "The Skyline Problem", "Count of Smaller Numbers After Self" или "Data Stream as Disjoint Intervals", могут быть решены с использованием `TreeMap` или `TreeSet` (или аналогичных структур, таких как деревья отрезков/деревья Фенвика для некоторых случаев).
* Когда простой `heap` (очередь с приоритетом) не предлагает достаточной гибкости (например, вам нужно удалять произвольные элементы, а не только мин/макс, сохраняя при этом порядок).

Хотя стандартная библиотека Python не имеет прямого `TreeMap`/`TreeSet`, понимание их принципов имеет решающее значение для эффективного решения многих алгоритмических задач, и вы можете использовать их в других языках или полагаться на `sortedcontainers` для соревновательного программирования на Python.
